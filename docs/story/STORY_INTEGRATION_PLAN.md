# 剧情接入规划

本文把主线剧情拆成可执行工程任务，说明每个阶段应该接入哪些内容，以及如何避免剧情逻辑污染核心状态机和 UI 绘制代码。

## 当前代码状态判断

当前工程已经具备剧情接入所需的主要挂点：

- `GameSession`：十日周期、阶段状态机、玩家属性、行动结果、每日总结和结局字段。
- `DayContext`：当前已有天气和事件字段，可作为后续日提示或玩法上下文来源。
- `ActionResult.summary`：地点行动摘要入口。
- `last_summary`：每日总结入口。
- `main_ending` / `final_summary`：结局文本入口。
- `save_game`：已保存摘要和结局文本，可以恢复阶段边界。
- `game_flow`：已有标题、地图、地点、每日总结、结局页面。

主要风险：

- 运行时文案当前分散在核心和 UI 中，继续扩写会造成硬编码扩散。
- 有存档时应用启动会直接进入游戏会话，可能跳过标题页和开场剧情。
- 字体字形清单需要覆盖新增剧情文本，否则会出现缺字或误判。
- 如果为了剧情新增路线统计或 NPC 状态，会触及存档格式，必须拆成单独 issue。

## 推荐架构

P1 先新增 raylib-free 剧情层：

- `src/core/story_text.hpp`
- `src/core/story_text.cpp`
- `tests/story_text_test.cpp`

最小接口方向：

```cpp
const char* opening_story();
const char* daily_prompt(int day);
const char* location_result_summary(Location location, ActionOutcome outcome);
const char* council_opening();
const char* placeholder_ending_text();
std::string story_glyphs();
```

P1 不需要立刻做外部数据加载器。先把 S 级文本集中在核心层，保证测试和 UI 接入简单。P4 若文本量扩大，再评估 `assets/data/story/` 数据文件。

## P1：MVP 主线骨架

新增 issue：`19-p1-mvp-story-shell.md`。

目标：

- 修正标题页/存档恢复流程，保证普通启动先进入标题页。
- 在标题页或新游戏后展示开场文本。
- 地图页展示十天每日提示。
- 模拟地点结果使用剧情短摘要。
- 每日总结页展示行动摘要和生活收束。
- 第十天结局前展示评议会开场，或合并进结局页顶部。
- 占位结局使用“平凡小镇新人”文案。
- 新增剧情文本字形覆盖检查。

不做：

- NPC 状态机。
- 好感度。
- 对话树。
- 任务链。
- 旧夏日集市完整线。
- 需要修改存档格式的路线统计。

## P2：白天地点回声

P2 的 09/10/11 不需要单独新建剧情 issue，而是在各地点 issue 中补充验收：

- 地点规则输出可解释的玩法统计。
- 地点结束只生成一个统一 `ActionResult`。
- `ActionResult.summary` 使用剧情层或地点摘要映射生成。
- UI 不自行拼接长剧情。
- 文案符合 [`STYLE_GUIDE.md`](STYLE_GUIDE.md)。

## P3：酒馆与正式结局

P3 的 12/15 承担剧情正式收束：

- Issue 12：酒馆夜晚摘要、输赢回声、回家/酒馆的夜晚选择差异。
- Issue 15：第十天评议会、七个正式主结局、最终状态、成长路线和触发依据。

若结局判定需要路线统计，应在 Issue 15 中集中加入并更新存档测试。

## P4：内容扩充与润色

P4 的 16/17 承担人工打磨：

- 扩写 NPC 三段式文本。
- 决定旧夏日集市碎片是否进入最终版本。
- 根据试玩反馈压缩文本长度。
- 验证 960×540 下文本可读性。
- 固化最终文案和数值阈值。
- 更新演示脚本、PPT 和署名材料。

## 接入顺序

1. 先完成 P1 剧情骨架，集中 runtime 文案入口。
2. 再接 P2 白天地点，地点只负责统计和结果，不直接改全局剧情状态。
3. 再接 P3 酒馆与正式结局，集中处理路线统计和唯一主结局。
4. 最后 P4 扩写、平衡和人工批准。

## 验收原则

- 文案接入必须有自动化测试覆盖核心选择逻辑。
- UI 截图只能证明显示效果，不能代替剧情选择逻辑测试。
- 存档恢复后应显示同一阶段文本，不重新触发已结算剧情。
- 新增剧情文本必须进入字形覆盖。
- 任一文案不得与当前 PRD、`CONTEXT.md`、`DESIGN.md` 的核心约束冲突。
