# UI 与视觉素材协作策略

本文定义 P2/P3 期间 NPC UI、场景 UI 和 imagegen 视觉素材的协作方式。目标是让组员优先完成可测试的地点逻辑，同时保证最终视觉风格可以在 P4 统一收口。

## 总策略

当前阶段采用三层策略：

1. imagegen 先定统一视觉方向和少量稳定主资产。
2. P2/P3 地点模块允许使用占位 UI 完成逻辑、测试和统一行动结果。
3. P4 再统一替换最终场景、NPC、面板、教程、结算和结局表现。

结论：imagegen 不阻塞地点逻辑开发；占位 UI 是 P2/P3 的合法交付方式，但必须保持布局可替换、逻辑可测试、结果统一。

## 现在适合使用 imagegen 的内容

P1/P2 初期可以生成并归档以下候选资产，用于统一风格参考和后续替换：

| 类型 | 用途 | 当前定位 |
| --- | --- | --- |
| 场景背景 | 餐馆、便利店、图书馆、酒馆、家等 960×540 室内或夜晚页面基调 | 候选资产，不阻塞地点逻辑 |
| NPC 头像或半身像 | 镇长、餐馆老板、便利店店主、图书馆管理员、酒馆老板或常客 | 候选资产，P4 前可不接入 |
| UI 面板风格板 | 对话框、结算卡片、按钮、标签框、教程面板 | 风格参考，最终需统一实现 |
| 标题/地图补充图 | 现有标题页和地图页的风格延展 | 可在 P1/P4 间逐步替换 |

所有 imagegen 资产进入正式交付前必须保留：

- 原始生成图。
- 处理后图。
- 生成用途说明。
- 文件 hash。
- 人工验收截图或记录。

## P2/P3 允许的占位 UI

地点负责人可以使用以下占位形式完成模块：

- 程序绘制背景色、面板和按钮。
- 占位 NPC 头像框。
- 简单图标或 Kenney 点缀素材。
- 纯文本说明、统计表和结果卡片。
- 临时色块或线框布局。

占位 UI 必须满足：

- 使用 960×540 逻辑画布。
- 中文文字清晰可读。
- 鼠标热区和键盘操作可验证。
- 必要玩法信息完整显示。
- 不把最终美术尺寸写死进规则层。
- 不让地点规则依赖图片是否加载成功。
- 不在 UI 绘制代码中硬编码长剧情文案。

## 组员开发边界

P2/P3 各模块优先级如下：

1. 规则引擎可脱离 raylib 测试。
2. 固定种子可复现。
3. UI 能展示必要状态并发送用户意图。
4. 模块结束只返回一个统一 `ActionResult`。
5. 行动摘要接入 `StoryText` 或后续剧情文本入口。
6. 占位 UI 保持布局可替换。

不得为了临时视觉效果破坏以下边界：

- 地点规则依赖 raylib。
- UI 直接修改玩家状态、库存、天数或酒馆战绩。
- 各模块私自定义一套最终按钮、字体、NPC 框或结算页风格。
- 把 imagegen 资源缺失变成逻辑不可运行。
- 未经评审把候选素材当成最终交付素材。

## P4 统一收尾范围

P4 由展示/美术集成负责人统一处理：

- 替换各地点场景背景。
- 替换 NPC 头像或半身像。
- 统一按钮、面板、状态栏、弹窗和教程框。
- 统一每日总结、评议会和结局页布局。
- 检查 960×540 下中文可读性。
- 更新 `CREDITS.md`、资源 hash、许可证和素材来源记录。
- 记录人工视觉、可读性、音频和教程验收结论。

## 验收口径

P2/P3 issue 通过不要求最终美术完成，但要求：

- 玩法逻辑和规则测试完成。
- UI 可操作、可理解、不卡流程。
- 占位资源不会污染最终素材规范。
- 后续替换最终视觉时不需要重写规则层。

P4 issue 通过才要求：

- 最终视觉风格统一。
- imagegen、Kenney、字体等资源来源和许可完整。
- NPC、场景和 UI 组件完成最终人工验收。
