# 团队开发规范

本文定义《像素小镇：十日经营计划》的五人协作、分支、评审、测试和交付规范。它补充 `AGENTS.md`、PRD、设计文档和本地 issue，不替代具体 issue 的验收标准。

## 基本原则

- `main` 始终保持可构建、可测试、可演示到当前阶段门槛。
- 每次开发只围绕一个本地 issue 或一个明确修复点展开。
- 先完成可运行的纵向闭环，再补内容数量和表现细节。
- 规则逻辑优先可测试，UI 只负责展示状态和发送用户意图。
- 任何人都不能在未同步的情况下改变共享契约、存档格式、行动结果结构、随机源语义或阶段状态机。
- 临时数值可以用于跑通切片，但不能被当作最终平衡结论。

## 角色与职责

| 角色 | 主责 | 必须同步的对象 |
| --- | --- | --- |
| 核心与集成负责人 | 状态机、统一行动结果、存档、结局、CI、跨模块集成 | 所有地点负责人 |
| 餐馆负责人 | 餐馆规则、UI 接入、内容和测试 | 核心与交互外壳负责人 |
| 便利店负责人 | 商品、库存、需求模型、经营 UI 和测试 | 核心与存档负责人 |
| 图书馆负责人 | 题库、分类匹配、图书馆 UI 和测试 | 核心与资源验证负责人 |
| 酒馆与展示负责人 | 酒馆外壳、五子棋、骗子骰子、视觉原型和演示 | 核心与交互外壳负责人 |

负责人不等于唯一开发者。每个模块可以由多人协作，但必须有一名成员对 issue 验收、测试证据和合并质量负责。

## 开发流程

1. 选择一个 `Status: ready-for-agent` 或已人工批准的 `Status: ready-for-human` issue。
2. 阅读该 issue 的 `Blocked by`，确认真实依赖已完成。
3. 阅读最小必要文档：`AGENTS.md`、`CONTEXT.md`、`docs/README.md`、目标 issue，以及相关设计段落。
4. 在短期 feature branch 上工作，不直接向 `main` 提交。
5. 先实现或更新规则测试，再接入 UI、资源或演示路径。
6. 完成 issue 的全部验收标准，并记录未验证项和原因。
7. 请求至少一名非作者成员审查；涉及共享契约时必须让核心与集成负责人参与。
8. 合并后确认 `main` 仍可构建、测试和运行到当前阶段门槛。

## 分支与提交

人工成员的分支名使用短小、可追踪的形式：

- `p0/01-offline-baseline`
- `p1/04-one-day-loop`
- `p2/10-store-inventory`
- `fix/save-corrupt-input`
- `docs/development-guide`

自动化代理使用 `codex/` 前缀，后续部分保持同样的可追踪结构，例如 `codex/p0-01-offline-baseline`。不要复用其他成员正在使用的分支名。

提交信息使用简洁动词短语，推荐格式：

```text
area: summary
```

示例：

```text
core: reject duplicate action results
store: persist unsold inventory
docs: add team development guide
test: cover day ten ending boundary
```

一次提交应表达一个清晰意图。不要把无关格式化、素材替换、规则改动和文档改动混在同一个提交里。

## 代码组织约束

实现阶段开始后，目录命名应尽量保持以下边界：

| 目录 | 用途 |
| --- | --- |
| `src/core/` | 游戏会话、阶段状态机、玩家状态、行动结果、结局、随机源 |
| `src/locations/` | 餐馆、便利店、图书馆和酒馆规则 |
| `src/app/` | raylib 应用装配、场景切换、输入适配 |
| `src/ui/` | 共享 UI 控件、逻辑画布、像素缩放、文本绘制 |
| `src/io/` | 存档、资源验证、日志、演示预设加载 |
| `assets/` | 实际使用的字体、图片、音频和数据文件 |
| `tests/` | doctest 测试，不创建图形窗口的规则测试优先 |
| `third_party/` | 锁定版本的 raylib、doctest 和对应许可证 |

具体目录可在 P0 Issue 01 中微调，但必须保持“规则层不依赖 raylib”的边界。

## C++ 规范

- 使用 C++17，不依赖编译器私有扩展。
- 领域层使用标准库类型和项目自定义类型，不暴露 raylib 类型。
- 避免全局可变状态；随机性通过显式种子或随机源传入。
- 共享数据结构优先小而明确，字段含义写在类型名、成员名或测试中。
- 状态修改集中在核心系统，地点模块只返回行动结果。
- 配置值集中管理，禁止在规则分支和 UI 中散落不可追踪的魔法数字。
- 错误路径要返回可诊断信息，不能静默失败或只在控制台输出。

## 测试与验证

每个 issue 的默认完成要求：

- 运行与改动相关的 doctest/CTest。
- 新增或修改规则行为时补充自动化测试。
- 涉及随机性时使用固定种子验证复现。
- 涉及存档时覆盖正常往返、损坏输入和版本不兼容。
- 涉及 UI、像素缩放、字体、音频或现场演示时记录人工验收结果。
- 合并前检查构建说明、测试结果和未验证风险是否写清楚。

在构建系统创建前，不能声称已完成编译或测试验证；只能说明当前仓库尚无可执行检查。

## 评审规则

评审优先看风险，不只看代码风格：

- 是否破坏阶段状态机、行动结果、存档或随机复现。
- 是否让地点规则依赖 raylib、UI 或不可测试的全局状态。
- 是否绕过本地 issue 的验收标准。
- 是否缺少关键边界测试。
- 是否引入范围外功能、额外依赖或不可追踪素材。
- 是否让 `main` 难以从干净检出构建。

涉及以下内容时必须暂停并同步团队：

- 新增或替换第三方依赖。
- 改变存档格式、目录结构或发布包写入位置。
- 改变玩家状态字段、行动结果字段或阶段枚举。
- 新增地点、属性、结局、小游戏或大范围 UI 系统。

## 每日协作节奏

建议每天进行一次短同步，内容控制在以下四项：

- 昨天完成了哪个 issue 的哪条验收标准。
- 今天准备推进哪个最小切片。
- 当前阻塞是否影响其他人。
- 是否改变了共享契约、数据文件、资源规范或构建命令。

同步记录只保留重要决策和阻塞，不把日常流水账写进稳定文档。

## 完成定义

一个 issue 只有在满足以下条件后才能视为完成：

- 所有验收标准已完成，或明确记录未完成项并获得团队接受。
- 相关自动化测试通过，人工验收项有记录。
- 文档、构建说明或资源清单已随行为变化更新。
- 没有遗留调试开关、无界日志、大文件、临时素材或个人路径。
- 至少一名非作者成员完成审查。
- 合并后 `main` 仍满足当前阶段门槛。
