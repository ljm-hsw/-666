# 《像素小镇：十日经营计划》产品需求文档

Status: ready-for-agent

## Problem Statement

五人 C/C++ 实训团队需要完成一个规模可控、便于并行开发、现场展示效果明确，同时具有完整产品闭环的游戏项目。

如果把餐馆、商店、图书馆和桌游分别实现为互不相关的小游戏，玩家缺少持续目标，团队模块之间也缺少统一接口，最终成果容易成为功能拼接。反之，如果直接模仿大型开放世界生活模拟游戏，地图、内容、美术和交互成本会超过实训周期。

项目需要在有限周期内解决以下问题：

- 用统一目标和状态连接不同玩法。
- 让玩家的每日选择产生可理解的机会成本和最终反馈。
- 让五名成员能够并行开发，同时维持可集成、可测试的模块边界。
- 控制内容规模，优先完成可从开局玩到结局的闭环版本。

## Solution

构建一款单人、桌面端、像素风生活模拟经营游戏。玩家扮演刚来到小镇的新居民，在固定十个游戏日内，每天选择一个白天工作和一个夜晚活动，管理金钱、体力、声望、知识和心情。

白天可进入餐馆、便利店或图书馆：餐馆强调反应和准确性，便利店强调进货与定价判断，图书馆强调知识分类与服务。夜晚可进入酒馆挑战五子棋或骗子骰子，也可回家休息。所有模块通过统一的行动结果影响同一份玩家状态。

第十天结算后，系统依据玩家的综合表现触发稳定、可解释的最终结局。小镇地图只负责地点选择和氛围呈现，不建设大型开放世界。

### 核心体验目标

- 玩家能够理解“今天选择什么，就放弃了什么”。
- 每个地点提供不同节奏和收益倾向，但始终服务于十日经营目标。
- 一局能够在适合现场演示的时间内完成或快速跳转到结局展示。
- 玩家能从状态变化和最终结局中理解自己形成的成长路线。

### MVP 验收标准

- 可以从新游戏开始，连续完成十个游戏日并进入一个最终结局。
- 每天只能完成一个白天工作和一个夜晚活动，阶段不能重复或跳过。
- 餐馆、便利店、图书馆、酒馆和回家休息均可进入、完成并返回统一日程。
- 五项通用属性和酒馆战绩能够随行动正确变化，并在界面上得到反馈。
- 餐馆至少提供订单、计时、正确率和收益结算闭环。
- 便利店至少提供进货、定价、需求提示和销售结算闭环。
- 图书馆至少提供读者需求、类别匹配和奖励结算闭环。
- 酒馆至少提供可完成的五子棋和骗子骰子，以及胜负收益结算。
- 各阶段边界能够自动保存；重新启动后能够恢复同一局的核心状态。
- 第十天结束后能够按照稳定优先级产生已定义结局之一。
- 第十天仍完成完整白天和夜晚流程，剩余库存清算后只产生一个主结局。
- 在目标演示环境中可完成构建和启动，不依赖网络服务。

## User Stories

1. 作为新玩家，我希望从简短背景介绍开始游戏，以便理解自己为什么来到小镇以及十日目标。
2. 作为新玩家，我希望看到当前天数和剩余天数，以便规划整局节奏。
3. 作为玩家，我希望随时看到金钱、体力、声望、知识和心情，以便做出行动选择。
4. 作为玩家，我希望每天先选择一个白天工作，以便形成明确的日程节奏。
5. 作为玩家，我希望完成白天工作后再选择夜晚活动，以便感受到一天的推进。
6. 作为玩家，我希望系统阻止重复进行同一阶段，以便规则公平且不会无限刷取收益。
7. 作为玩家，我希望地图集中展示家、餐馆、便利店、图书馆和酒馆，以便理解这些玩法属于同一个小镇。
8. 作为玩家，我希望锁定或不可进入的地点给出原因，以便知道是阶段、体力还是其他条件限制。
9. 作为玩家，我希望每次行动结束后看到属性变化摘要，以便理解选择造成的结果。
10. 作为玩家，我希望每日结束时看到当天总结，以便调整次日计划。
11. 作为餐馆玩家，我希望看到顾客订单，以便按要求完成点餐或上菜。
12. 作为餐馆玩家，我希望订单存在合理时间压力，以便获得高节奏体验。
13. 作为餐馆玩家，我希望正确服务顾客获得金钱和声望，以便操作表现影响成长。
14. 作为餐馆玩家，我希望错误或等待过久降低评价，以便准确性和速度都有意义。
15. 作为餐馆玩家，我希望结算展示服务数量、正确率和收益，以便复盘表现。
16. 作为便利店经营者，我希望查看可采购商品及成本，以便制定进货计划。
17. 作为便利店经营者，我希望决定各商品进货数量，以便在预算和库存风险间权衡。
18. 作为便利店经营者，我希望设置商品售价，以便在利润率和销量间权衡。
19. 作为便利店经营者，我希望在经营前看到天气或事件提示，以便预测需求。
20. 作为便利店经营者，我希望不同商品对天气和事件有不同需求反应，以便经营判断有实际作用。
21. 作为便利店经营者，我希望结算展示销量、库存、收入、成本和利润，以便理解经营结果。
22. 作为便利店经营者，我希望未售库存跨日保留，以便形成连续的十日经营策略。
23. 作为图书馆工作人员，我希望看到读者的文字需求，以便判断其需要的书籍类别。
24. 作为图书馆工作人员，我希望在多个书籍类别中做出选择，以便体现知识匹配玩法。
25. 作为图书馆工作人员，我希望正确推荐提高知识和声望，以便获得区别于赚钱工作的成长。
26. 作为图书馆工作人员，我希望错误推荐获得清晰反馈，以便学习分类规则。
27. 作为图书馆工作人员，我希望任务节奏比餐馆舒缓，以便不同地点具有明显体验差异。
28. 作为夜晚玩家，我希望可以选择回家休息，以便稳定恢复体力并规避酒馆风险。
29. 作为夜晚玩家，我希望可以进入酒馆，以便用风险换取金钱、心情或特殊成绩。
30. 作为五子棋玩家，我希望规则合法性、轮次和胜负判断正确，以便游戏公平可完成。
31. 作为五子棋玩家，我希望能够与本地对手完成一局，以便获得完整桌游体验。
32. 作为骗子骰子玩家，我希望能够叫点、质疑并看到骰子揭示，以便理解每轮结果。
33. 作为骗子骰子玩家，我希望电脑对手遵循同一套公开规则，以便结果可信。
34. 作为酒馆玩家，我希望胜负影响金钱、心情和酒馆战绩，以便夜晚选择与最终结局关联。
35. 作为谨慎玩家，我希望体力不足时系统阻止高消耗行动并说明原因，以便避免非法状态。
36. 作为规划型玩家，我希望不同地点具有不同收益与消耗特征，以便形成赚钱、声望、知识、娱乐或均衡路线。
37. 作为玩家，我希望随机事件在不同游戏中有所变化，以便提高重复游玩价值。
38. 作为测试者，我希望同一随机种子产生相同事件和结果，以便稳定复现问题。
39. 作为中途退出的玩家，我希望关键阶段完成后自动保存，以便下次继续当前进度。
40. 作为返回玩家，我希望读取存档后恢复天数、阶段、属性和必要战绩，以便不会丢失进度或重复领取收益。
41. 作为玩家，我希望新开游戏时明确提示是否覆盖已有存档，以便避免误操作。
42. 作为完成十天的玩家，我希望看到与自身表现对应的结局，以便感受到选择产生结果。
43. 作为完成十天的玩家，我希望结局说明关键判定依据，以便理解为什么得到该结果。
44. 作为追求金钱的玩家，我希望有机会获得“冷酷赚钱机器”结局，以便高金钱低声望路线得到反馈。
45. 作为追求综合表现的玩家，我希望有机会获得“小镇明星经营者”结局，以便均衡高成就得到认可。
46. 作为重视人际服务的玩家，我希望有机会获得“受欢迎的普通居民”结局，以便高声望路线得到认可。
47. 作为重视知识的玩家，我希望有机会获得“图书馆之星”结局，以便知识路线得到认可。
48. 作为酒馆挑战者，我希望有机会获得“酒馆传说”结局，以便桌游表现进入长期反馈。
49. 作为普通玩家，我希望中等表现也有“平凡小镇新人”结局，以便游戏不会只有成功或失败两种评价。
50. 作为经营失利的玩家，我希望看到“经营失败者”结局及关键原因，以便理解失败而不是遭遇突然惩罚。
51. 作为现场演示者，我希望能够快速开始一局并清楚展示核心循环，以便观众在短时间内理解产品。

60. 作为开发成员，我希望首次构建不依赖下载外部依赖，以便团队和演示机器在离线环境中可靠构建。

## Implementation Decisions

### 产品与平台边界

- MVP 是桌面端、单机、单人 C++17 应用，运行时不依赖网络服务。
- 最终演示目标为 Windows 10/11 x64，标准编译环境为 Visual Studio 2022 与 MSVC。
- CMake 是唯一构建入口；在不削弱 Windows 交付稳定性的前提下，保持 macOS 本地开发可用。
- 首个交付版本以完整十日闭环为优先，不以内容数量或开放世界规模为目标。
- 图形、输入和音频统一使用 raylib 6.0；自动化测试统一使用 doctest 2.5.2。
- raylib 的锁定版本源码和 doctest 单头文件保存在仓库 `third_party/`，并保留对应许可证。首次配置与构建不得访问网络，也不得依赖用户全局安装。
- 不引入 SDL、SFML、raygui 或第二套图形、音频、输入、UI 框架。
- 小镇地图只负责地点入口、状态展示和氛围，不实现角色移动、碰撞或大型场景探索；仅图书馆和酒馆的固定室内场景允许有限移动与静态碰撞。

### 交付与评审优先级

- 项目优先级固定为：完整十日闭环、稳定现场演示、清晰模块边界、关键规则自动化测试、额外内容数量。
- 最终演示前至少七天冻结功能；冻结后只处理缺陷、平衡、资源替换和演示材料。
- 交付物包括源码、可复现构建说明、Windows 可执行包、自动化测试结果、演示预设和演示 PPT。
- 五名成员必须能够使用同一套 CMake 配置完成构建；个人 IDE 配置不得成为项目的隐式依赖。

### 核心状态机

- 游戏阶段的产品流程固定为：日初、白天工作、夜晚活动、每日结算、最终结局；实际可持久化状态枚举和存档边界以 `docs/DESIGN.md` 的状态机为准。
- 核心系统是天数、阶段和玩家状态的唯一写入者。
- 每个地点模块接收只读的当前状态与当日上下文，完成后返回统一行动结果；地点模块不得直接推进时间或修改全局状态。
- 行动结果包含通用属性变化、模块战绩、结果摘要和是否正常完成。核心系统必须保证同一行动结果只应用一次。
- 金钱可以因酒馆风险减少；体力、声望、知识和心情必须有明确下限。具体平衡上下限集中定义，不散落在 UI 或地点逻辑中。
- 游戏不会因属性触底提前结束。即使金钱或体力不足，玩家仍必须存在至少一条无需前置资源的恢复路线；经营失败只在第十天结局中体现。
- 心情必须对工作结果产生可见、可解释的修正，但修正区间、阈值和倍率属于待调优的数值平衡参数。
- 声望和知识在 MVP 中用于成长反馈与最终结局，不额外修改顾客数量、商品价格或地点难度。

### 数值平衡策略

- 当前 PRD 固定属性含义、变化方向和结算责任，不固定具体初始值、上下限、消耗、恢复、收益、价格、概率、计时、赌注或结局阈值。
- 所有数值平衡参数由核心层集中提供，地点模块通过明确输入读取；禁止在界面或规则分支中散落不可追踪的魔法数字。
- 每个阶段在纵向闭环开发时提供一组可运行的临时基线值，但这些值不构成最终产品承诺。
- 正式取值在阶段 4 通过固定种子模拟、自动化边界测试和人工试玩共同确定。
- 首版只要求开发期集中配置，不建设面向玩家或策划的运行时数值编辑器。

### 随机性与日程上下文

- 天气、事件、顾客或读者需求由当日上下文提供，而不是由各界面任意生成。
- 所有随机逻辑使用可注入或可记录的随机种子。正常游戏可生成新种子，测试和演示可使用固定种子。
- 随机结果影响选择条件和收益，但不得使任何一天无法完成。
- 便利店需求使用可解释的组合模型：基础需求、价格档位、天气/事件修正和受固定种子控制的有限随机波动；具体倍率属于数值平衡参数。

### 核心系统模块

- 负责新游戏、十日流程、阶段校验、玩家状态、行动结果应用、每日结算、存档和结局判定。
- 对其他模块提供少量稳定能力：读取当前状态、查询可用行动、提交一次行动结果、保存或恢复游戏。
- 结局判定使用集中配置的条件与明确优先级；禁止把结局判断分散到地点模块。

### 餐馆模块

- MVP 采用订单队列、有限等待时间、点击菜品、提交上菜、正确性判定和局末结算，不实现自由移动或复杂烹饪步骤。
- 收益由完成量、正确率和等待表现共同决定；反复错误或超时会降低声望收益。
- 错单提交后被拒绝并造成时间损失，超时顾客离开；这些模块内事件统一汇总后再生成行动结果。
- 餐馆模块不负责直接修改金钱或声望，只返回结算后的行动结果。

### 便利店模块

- MVP 商品集合保持小而有差异，覆盖日常品、雨天相关商品、学习相关商品和休闲食品等需求类别。
- 玩家在固定预算内选择商品和数量，并为商品选择低价、标准价或高价档位，然后根据当日上下文完成一次销售模拟。
- 进货前显示天气与事件提示，但不展示准确销量或唯一正确答案。
- 销售结算必须显示成本、销量、收入、利润和剩余库存。
- 未售库存跨游戏日保留；MVP 不实现保质期、自然损耗或仓储容量系统。
- 第十天夜晚活动结束后，剩余库存按集中配置的折价比例自动转换为现金，再进入最终结局判定。
- 需求计算应封装为独立、可测试模块，只接收商品、价格和当日上下文并返回销量结果。

### 图书馆模块

- MVP 使用有限、清晰的书籍类别和读者需求题库，不把玩法改为逐个书架搜索或文字输入；允许玩家在固定室内场景中靠近管理员并进入现有分类答题流程。
- 书架、家具、柜台和墙体属于不可穿越的静态障碍；后续新增或替换这些场景物件时必须同步维护碰撞范围。
- 书籍类别和读者需求保存在仓库数据文件中，由规则引擎读取，不硬编码在界面代码里。
- 读者需求映射到一个正确类别；玩家从候选类别中选择后立即获得正确或错误反馈。
- 回合结束依据正确数和连续正确表现计算知识、声望及少量金钱奖励。
- 题库内容与匹配规则分离，便于独立扩充和测试。

### 酒馆模块

- MVP 只包含五子棋和骗子骰子，不加入第三种卡牌游戏。
- 每晚只能选择并完成其中一种酒馆挑战，不能在同一晚重复挑战或切换到第二种游戏。
- 五子棋采用玩家对电脑的自由五子棋规则，不实现禁手；电脑按“立即获胜、立即防守、位置评分”的确定性启发式决策，不实现高深度 Minimax 或多难度等级。
- 骗子骰子采用玩家对一个电脑对手的 1v1 淘汰制：双方隐藏骰子并轮流提高叫点或质疑，每轮失败方失去一枚骰子，先失去全部骰子者输。
- 骗子骰子中的一点通常作为万能点；叫一点时不再作为其他点数的万能点。初始骰子数量属于数值平衡参数。
- 两种酒馆规则引擎均负责合法性、轮次和胜负判断，并与显示、输入和电脑策略分离。
- 电脑对手以“规则正确、可完成、可调难度”为目标，不要求高级 AI。
- 酒馆开始前选择低、中、高赌注档位，赌注不能超过当前金钱；平局退还赌注。
- 酒馆结算返回金钱、心情和战绩变化；具体赌注金额由数值平衡参数提供。

### 地图、界面与反馈

- 主界面必须持续展示当前天数、阶段和通用玩家状态。
- 小镇地图采用静态像素插画和可点击地点热点，不控制角色在地图中行走。
- 图书馆和酒馆可以使用固定镜头室内场景；主角仅使用上下左右方向键移动，不绑定 WASD，鼠标仍可作为互动入口的等价方式。
- 室内移动采用场景内明确配置的可行走边界与静态碰撞体。主角和巡逻 NPC 均不得进入家具、书架、吧台、桌椅或墙体占据的范围。
- 对话打开时冻结主角移动、NPC 轨迹和地点计时；关闭后回到同一室内位置继续。
- 地图依据当前阶段展示可进入地点；不可用地点必须给出原因。
- 所有地点通过相同的进入、完成、取消和结果展示流程接入核心系统。
- 应用使用单一窗口和显式场景状态机，不创建第二窗口。
- 场景范围固定为标题、新游戏/继续、小镇地图、三个白天地点、酒馆选择、五子棋、骗子骰子、每日总结、最终结局和暂停/设置。
- 内部逻辑画布为 960×540，默认窗口为 960×540。窗口变化时使用整数倍缩放与留黑边，不能拉伸或模糊像素画面。
- 界面只实现项目实际需要的按钮、面板、文本和焦点控件，不建设通用 GUI 框架。
- 主要交互使用鼠标，常用操作提供键盘快捷键；手柄和触屏不属于 MVP。
- 打开暂停菜单、窗口失焦或系统最小化时，所有地点计时和动画推进冻结；恢复焦点后继续。
- 视觉表现采用一致像素风，但逻辑验收不能依赖动画完成。
- MVP 仅提供简体中文。仓库内附带允许再分发的中文字体及许可证，不建立多语言资源系统。
- 每个地点首次进入时显示一页可跳过说明，暂停菜单允许重新查看；不制作独立教程关卡。
- 地点基础规则在十天内保持不变，仅通过数值平衡参数增加订单压力、需求不确定性或题目难度，不使用根据玩家表现自动变化的动态难度。
- 演示模式通过演示预设或固定种子实现，不在正式规则中加入作弊入口。

### NPC 与内容边界

- 顾客和读者只属于当前地点任务，不建立持久姓名、好感度、跨日记忆或个人剧情。
- 已命名的图书馆管理员和酒馆常客可以在对应固定室内场景使用可复现的作者轨迹；位置在每次进入场景时重置，不写入 v1 存档，也不引入通用寻路或动态避障。
- 开场和结局可以使用简短旁白，但不建设对话树或任务系统。
- 商品、订单、读者需求、天气和事件以数据内容扩充；具体数量属于阶段 4 内容与平衡工作。

### 美术与资源

- 美术、字体和音频必须由团队原创，或具有允许项目再分发的明确许可证；禁止提取或改编商业游戏素材。
- 仓库维护 `CREDITS.md`，记录所有第三方资源名称、作者、来源、版本和许可证，并随发布包提供许可证文本。
- 全部像素资源遵循统一调色板、像素网格和命名约定；tile 与 sprite 尺寸在首个视觉原型验收后锁定。
- 小型 PNG、字体和 OGG 直接纳入 Git。只有出现明显大文件问题时才评估 Git LFS，不预先引入。
- 启动时验证必要字体、贴图和数据文件。必要资源缺失时显示明确错误页、写入日志并停止进入游戏主循环。

### 音频

- raylib 负责全部音频加载与播放，不引入独立音频中间件。
- MVP 包含一首可循环背景音乐，以及地点切换、操作成功、操作失败和结算等关键音效。
- 设置只提供全局静音开关；不实现音乐、音效分别调节或复杂混音。
- 音频资源必须有明确再分发许可；音频缺失不得阻塞核心游戏流程。

### 存档

- 在日初、白天行动完成、夜晚行动完成和每日结算后自动保存，避免重复应用行动收益。
- 存档至少包含格式版本、随机种子、当前天数、当前阶段、玩家状态和酒馆战绩。
- MVP 只维护一个自动存档；开始新游戏前必须明确提示覆盖已有进度。
- 存档使用带版本号的行式纯文本格式，只依赖 C++ 标准库，不增加 JSON 依赖，也不直接序列化原始内存结构。
- 应用/发布目录旁的 `saves/slot1.sav` 保存自动存档，使解压目录可以整体移动；构建说明必须要求从用户可写目录运行。
- 只保存恢复游戏必需的领域状态，不保存可重新计算的 UI 临时状态。
- 读取失败或版本不兼容时应保留原文件并给出明确错误，不静默重置。
- 进入地点模块前可以返回地图且不消耗行动；模块开始后主动放弃会消耗该阶段且不产生收益。
- 程序异常关闭时恢复到最近阶段边界，并使用相同随机种子重新开始未完成模块；不保存小游戏内部的中间帧状态。

### 演示预设与日志

- 演示预设是仓库内受版本控制的只读状态，通过命令行参数显式加载，不出现在普通菜单中。
- 演示预设与 `saves/slot1.sav` 中的正式自动存档隔离，加载和游玩都不得覆盖玩家进度。
- 每次启动覆盖 `logs/latest.log`，记录应用版本、资源验证、存档错误和关键阶段转换。
- 日志不得包含遥测、网络上传、用户个人信息或无界增长的历史文件。

### 五人协作与持续集成

- issue 和 PRD 继续使用 `.scratch/` 本地 Markdown；代码使用私有 GitHub 远程协作。
- `main` 必须保持可构建。每项任务使用短期 feature branch，经至少一名其他成员审查后合并。
- 核心与集成负责人维护共享契约并处理跨模块集成冲突，但不代替各模块负责人完成自测。
- GitHub Actions 在 Windows 和 macOS 上执行 CMake 配置、构建和 CTest；CI 不能替代最终 Windows 演示机器的手工验收。

### 最终结局

- 首版结局包括：小镇明星经营者、冷酷赚钱机器、受欢迎的普通居民、图书馆之星、酒馆传说、平凡小镇新人和经营失败者。
- 判定条件集中配置，并在正式平衡前通过模拟与试玩确定具体阈值。
- 每局只产生一个主结局，同时展示最终属性和成长路线总结，不叠加多个结局或成就。
- 多个条件同时满足时依次判断：经营失败、小镇明星经营者、单项路线结局、平凡小镇新人。
- 多个单项路线结局同时满足时，选择相对表现最突出的指标；归一化方法与阈值属于数值平衡参数。
- 结局界面展示结局名称、最终状态和触发该结局的主要依据。

### 五人职责边界

1. 核心与集成负责人：核心状态机、地图接入、统一行动结果、存档、结局和持续集成。
2. 餐馆负责人：餐馆规则、交互、内容与自动化测试。
3. 便利店负责人：商品模型、需求模拟、经营交互与自动化测试。
4. 图书馆负责人：需求题库、匹配规则、交互与自动化测试。
5. 酒馆与展示负责人：五子棋、骗子骰子、酒馆交互，并与核心负责人共同完成全局 UI 和现场演示整合。

模块负责人可以独立实现内部逻辑，但共享状态、行动结果、输入抽象、随机源和资源加载接口必须由团队共同确认后再并行开发。

## Testing Decisions

### 测试原则

- 自动化测试验证外部可观察行为和稳定契约，不绑定私有函数、内部容器或具体 UI 布局。
- 规则逻辑必须能够脱离图形窗口运行，避免把鼠标点击或渲染截图作为主要正确性证据。
- 涉及随机性的测试使用固定种子或受控随机源。
- 每个缺陷修复应先增加能够复现问题的测试，再实施最小修复。
- 当前仓库没有既有测试 prior art；统一使用仓库内锁定的 doctest 2.5.2，并通过 CTest 提供单一测试入口。
- 测试源文件与生产源文件分离，禁止在生产代码中嵌入测试用例。

### 必须自动化测试的模块

- 核心状态机：合法阶段转换、重复提交拒绝、十天推进和第十天后停止。
- 玩家状态与每日结算：属性变化、上下限、结果只应用一次和当天摘要。
- 结局判定：每个结局的代表性边界、冲突优先级和普通回退结局。
- 餐馆规则：订单正确性、超时、正确率和收益计算。
- 便利店需求：价格、天气、事件和库存对销量及利润的影响。
- 便利店持久库存：未售库存进入下一游戏日，结算和存档后数量一致。
- 最终库存清算：只在第十天完整日程结束后执行，并在结局判定前计入现金。
- 图书馆规则：读者需求到类别的正确匹配及奖励计算。
- 五子棋规则：合法落子、非法落子、横竖斜胜利、和局和轮次。
- 五子棋 AI：优先完成立即获胜，其次阻止对手立即获胜，再使用稳定位置评分。
- 骗子骰子规则：叫点递增、万能点例外、质疑、揭示、失去骰子和最终胜负。
- 存档：单存档覆盖提示、阶段边界保存读取往返、版本不兼容、损坏输入和原存档保护。
- 固定种子：相同输入和种子产生相同当日上下文与规则结果。
- 暂停行为：暂停、失焦和最小化期间计时值不变化。
- 资源验证：必要资源存在时正常启动，缺失或无效时进入明确错误页并写入日志。
- 演示隔离：演示预设加载前后正式自动存档字节保持不变。

### 集成与人工验收

- 自动化集成测试至少覆盖一条从新游戏到十日结局的无图形流程。
- 自动化集成测试覆盖每个地点返回行动结果并由核心系统应用的契约。
- 人工验收像素 UI、输入手感、动画、音效、文字可读性、窗口适配和现场演示流程。
- 每次里程碑演示前执行一次真实构建环境的全流程烟雾测试，并记录未通过项。

## Out of Scope

- 小镇主地图或开放世界自由移动、滚动镜头、通用寻路、复杂动态碰撞和大地图探索；图书馆与酒馆固定室内场景的方向键移动及静态矩形碰撞不在此列。
- 联网、多人合作或对战、账号、排行榜和云存档。
- 手柄、触屏、多窗口和多语言支持。
- 多存档槽、云存档、JSON 存档和小游戏中间状态恢复。
- 系统安装程序、注册表集成和写入系统级目录。
- 复杂 NPC 对话、好感度、恋爱、任务链和动态剧情系统。
- 背包、装备、制作、农业、房屋装修和完整经济生态。
- 成就系统、创意工坊、模组或通用内容编辑器。
- 第三种及更多酒馆小游戏；MVP 仅包含五子棋和骗子骰子。
- 高级机器学习或高强度搜索型游戏 AI。
- 大量天气、节日和随机事件内容；首版只实现足以验证经营差异的代表性集合。
- 移动端、Web、主机平台和跨设备同步。
- 超过十天的无限模式或二周目继承系统。
- 在核心闭环稳定前增加更多职业地点、属性或结局。

## Further Notes

### 分阶段交付计划

#### 阶段 0：工程基线与契约

- 在已确定的 Windows 10/11 x64、Visual Studio 2022、MSVC、C++17 和 CMake 基线上，接入仓库内锁定的 raylib 6.0 与 doctest 2.5.2。
- 落实 ADR-0001 的依赖、离线构建和框架隔离约束。
- 定义玩家状态、游戏阶段、当日上下文、行动结果和随机源的共享契约。
- 建立可重复、锁定依赖且不依赖演示现场网络的构建、测试和资源加载基线。
- 配置私有 GitHub 远程及 Windows/macOS CI，并验证五名成员均可从干净检出完成构建。

完成标准：空应用能够构建启动，核心规则测试能够无图形运行，五名成员使用同一套工具链。

#### 阶段 1：最小纵向闭环

- 实现新游戏、日程状态机、玩家状态、地图占位界面、回家休息、每日结算、存档和一个占位结局。
- 使用模拟地点结果跑通十天流程。

完成标准：即使地点内容尚未完成，也能从第一天推进到第十天、保存恢复并看到结局。

#### 阶段 2：三个白天模块并行接入

- 餐馆完成最小订单闭环。
- 便利店完成最小经营闭环。
- 图书馆完成最小匹配闭环。
- 每个模块先完成纯规则测试，再接入输入和显示。

完成标准：三个地点均可从地图进入、完成、返回并可靠改变统一玩家状态。

#### 阶段 3：酒馆与完整结局

- 完成五子棋规则与本地电脑对手。
- 完成骗子骰子规则与本地电脑对手。
- 接入赌注、心情和酒馆战绩。
- 实现全部首版结局及优先级测试。

完成标准：夜晚风险选择和七种结局均进入完整循环。

#### 阶段 4：内容、平衡与展示打磨

- 补充代表性商品、订单、读者需求、天气和事件。
- 通过固定种子模拟与人工试玩调整收益、体力消耗和结局阈值。
- 确认单局时长、各地点回合时长、属性边界及全部数值平衡参数。
- 统一像素视觉、音效、反馈、教程和结果界面。
- 完成 `CREDITS.md`、第三方许可证、只读演示预设与现场演示脚本。

完成标准：完整测试通过，目标机器可稳定构建运行，现场演示能够覆盖核心循环和最终结局。

### 风险与控制

- **集成过晚**：先完成纵向闭环和共享契约，再并行开发地点。
- **范围膨胀**：新增地点、系统或小游戏必须替换现有范围或进入 MVP 之后。
- **规则与 UI 耦合**：所有地点先实现可独立测试的规则引擎，再连接渲染和输入。
- **数值不可解释**：集中维护收益、消耗和结局阈值，并用结算界面解释关键变化。
- **随机问题难复现**：记录并允许输入随机种子，演示和测试使用固定种子。
- **团队环境不一致**：阶段 0 固化工具链并在目标演示机器验证。
