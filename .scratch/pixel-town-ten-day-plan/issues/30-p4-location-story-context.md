# P4：建立地点剧情上下文与存档迁移

Status: done
Milestone: P4
Type: AFK
User stories: 37, 38, 40, 55, 61, 73, 74

## What to build

为“地点首访教程 + 后续日常/意外事件”建立 raylib-free 领域契约。`GameSession` 保存五个地点各自的已完成访问次数；剧情目录根据地点、游戏日、`DayContext`、访问次数和会话种子确定性选择一个事件脚本。该切片只建立状态、存档迁移和选择器，不接入具体场景 UI，也不让剧情脚本直接改变奖励或玩法规则。

现有 v1 存档必须可以迁移：新增访问次数默认 0，其他字段保持一致；写出后使用新版本格式。当前对话行号、弹窗状态、NPC 动画和事件 UI 不保存。

## Acceptance criteria

- [x] 新增明确的地点剧情查询上下文，至少包含地点、游戏日、天气、日事件、该地点已完成访问次数和确定性种子；接口不依赖 raylib。
- [x] `GameSessionSnapshot` 与正式存档保存餐馆、便利店、图书馆、酒馆和家的已完成访问次数，非法负数或缺失地点被明确拒绝。
- [x] 只有统一 `ActionResult` 被首次成功应用且结果为完成时，对应地点访问次数增加一次；重复结果、只进大厅、返回地图和主动放弃不增加。
- [x] v1 存档可迁移到新格式，访问次数默认 0，天数、阶段、属性、库存、已应用结果和酒馆战绩不丢失；未知未来版本继续明确报错。
- [x] 事件选择优先级固定为：首访教程、十日回收、当日主线事件、确定性意外事件、普通熟悉对话。
- [x] 相同会话种子、游戏日、地点、访问次数和 `DayContext` 始终选择相同事件；重新打开大厅或读档不会重抽。
- [x] 选择结果包含稳定事件 ID、脚本触发/脚本内容和事件类别；纯查询不修改玩家状态或会话阶段。
- [x] 同一地点同一种意外事件一局最多选择一次所需的去重输入或规则被显式定义，不引入系统时间随机。
- [x] 首批事件只改变对话和提示；不得在本 issue 中修改客流、需求、题目、赌注、奖励或结局阈值。
- [x] 自动化测试覆盖五地点计数、重复结果幂等、放弃不计数、v1 迁移、损坏输入、选择优先级和确定性。
- [x] `docs/DESIGN.md`、构建/存档说明和测试基线同步更新。
